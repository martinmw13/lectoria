Lectoria

Multimodal EPUB reader with AI-driven narrative enrichment. A two-stage LLM pipeline produces a Narrative Context Map (NCM) that drives contextual music selection and image generation. Academic thesis — CEIA, FIUBA.

Python 3.14 FastAPI Google Gemini React + TypeScript Vite uv MTG-Jamendo

Overview

Lectoria enriches the EPUB reading experience with contextually matched music and AI-generated images that respond to the narrative. Every scene in a book gets classified by emotion, pacing, and scene type — that classification then drives which Jamendo track plays and which image appears.

The academic bet: LLM prompting alone (no fine-tuning) can extract narrative attributes rich enough that users perceive the enrichment as meaningfully better than random selection.

What it does

  • Parses EPUB files into numbered paragraphs
  • Runs a two-stage LLM pipeline to produce a Narrative Context Map (NCM)
  • Matches each scene to a music track from a curated Jamendo library
  • Generates or serves AI images per scene / chapter cover
  • Presents a paginated, scene-synchronized reader UI

What it does not do

  • No local model inference — all AI is BYOK (Bring Your Own Key)
  • No user accounts or cross-device sync
  • No audio generation (music is pre-recorded Jamendo tracks)
  • No PDF support
  • No mobile native app

NCM — The Central Artifact

The Narrative Context Map (NCM) is the JSON document produced by the pipeline and consumed by every other part of the system. It merges the outputs of two LLM stages into one file per book. Schema: lectoria/models/ncm.py.

NCM is the contract between modules. The pipeline writes it. The music service, image service, and reader UI all read it. Never derive narrative attributes at runtime — they are in the NCM.

Structure

NCM
├── book_map: BookMap          ← LLM 1 output (book-level)
│   ├── book_id, title, genre
│   ├── setting: { time_period, world, description }
│   ├── characters: Character[]
│   │   └── id (slug), name, aliases, physical_description, role, relationships
│   └── chapters: ChapterSummary[]
│       └── chapter_index, title, summary
│
└── chapters: ChapterAnalysis[]   ← LLM 2 output (per narrative chapter)
    ├── chapter_index
    ├── cover_description         ← image prompt for chapter cover art
    ├── llm_model, attempt_count, is_fallback   ← dev metadata
    └── scenes: Scene[]
        ├── scene_index
        ├── start_paragraph, end_paragraph    ← integer ranges, no gaps
        ├── characters_present: [character-id-slug]
        ├── emotion         ← primary music axis
        ├── pacing          ← secondary music axis
        ├── scene_type      ← tertiary music axis
        ├── setting: { location, time_of_day, weather }
        ├── image_prompt    ← ready-to-use prompt for image generation
        ├── transition_type
        ├── key_phrases, key_objects
        └── raw_emotion, raw_pacing, ...   ← original LLM values pre-coercion

The Pipeline

Triggered by POST /api/books/{book_id}/process. Streams progress via SSE. Takes 3–10 minutes per book. Implemented in lectoria/services/pipeline.py.

EPUB │ ▼ 1. Ingestion (lectoria/services/ingestion.py) │ Parse HTML from EPUB spine items │ Extract paragraphs (≥10 chars), chapter titles from headings │ Flag non-narrative chapters (table of contents, copyright, etc.) │ ▼ ChaptersData: chapters[] with numbered Paragraph[] │ ▼ 2. LLM 1 — Book Analysis (narrative.py: analyze_book) │ Full book text → single Gemini call │ Prompt: extract title, genre, setting, characters, chapter summaries │ Output validated as BookMap (Pydantic). Up to 3 retries. │ Intermediate saved: bookmap.json │ ▼ BookMap │ ▼ 3. LLM 2 — Chapter Analysis (narrative.py: analyze_chapter) │ Per narrative chapter — runs concurrently (semaphore=2) │ Each call: chapter text (numbered paragraphs) + BookMap context │ Prompt: segment into scenes with emotion, pacing, type, image_prompt, ... │ Coercion layer maps hallucinated enum values before Pydantic validation │ Up to 3 retries; fallback = single-scene covering all paragraphs │ ▼ ChapterAnalysis[] (one per chapter) │ ▼ 4. Assembly │ Reconcile characters discovered by LLM 2 but missing from BookMap │ Merge → NCM │ ▼ ncm.json saved to data/books/<book-id>/

Upload Flow (before pipeline)

POST /api/books/upload — the upload step runs ingestion, returns a cost estimate (chapter count + ~token count), and saves the EPUB and chapters.json to disk. Processing is a separate call so the user can review cost before committing API quota.

LLM Retry Strategy

  • Content retries (up to 3): handled in narrative.py — triggered by JSON parse failures, Pydantic validation errors, or missing required fields.
  • Rate-limit retries (up to 3): handled inside the provider in providers/llm/google.py — triggered by 429/RESOURCE_EXHAUSTED. Uses the API's retryDelay or exponential backoff starting at 30s.
  • Coercion layer: before Pydantic validation, ~40 common LLM-invented enum values are mapped to valid ones (e.g. "frustration" → "anger"). Original values preserved in raw_* fields.

Provider Protocol

All AI calls go through lectoria/providers/base.py protocols. Services never import provider SDKs directly.

class LLMProvider(Protocol):
    async def complete(prompt, *, system=None) -> CompletionResult
    def max_context_tokens() -> int

class ImageProvider(Protocol):
    async def generate(prompt, *, reference_image=None) -> bytes
    def supports_reference_image() -> bool

Current adapters: providers/llm/google.py (GeminiLLMProvider), providers/image/google.py (GeminiImageProvider). New providers = new module + call to register_llm_provider() or register_image_provider() in providers/registry.py.

Music Matching

Music selection is fully deterministic — no ML inference at read time. Implemented in lectoria/services/music.py. Source: MTG-Jamendo dataset.

Three-Phase Matching Pipeline

match_scene_to_track(scene, index, style="cinematic")
  │
  ├─ 1. Emotion filter
  │      candidates = [t for t in index if t.emotion_primary == scene.emotion]
  │
  ├─ 2. Style filter  (skip if style is None / "auto")
  │      candidates = [t for t in candidates if matches_preset(t, "cinematic")]
  │
  ├─ 3. Cosine ranking
  │      scene_vec  = one-hot encoding of scene emotion+pacing+scene_type tags
  │      track_vecs = pre-stacked matrix (N tracks × 59 tag dimensions)
  │      scores     = vectorized cosine similarity (numpy matmul)
  │
  └─ 4. Variety: skip if same as previous track (return ranked[1])

Fallback Chain

Ensures music always plays, even with sparse style coverage:

emotion + style → [empty?] → style only (all emotions) → [empty?] → emotion only

Hysteresis — When Does the Track Change?

Not every scene transition triggers a music change. Four emotion clusters determine "distance":

ClusterEmotions
Positivejoy, excitement, peace, wonder, romance
Darktension, anger
Melancholicsorrow
Neutralmystery
  • Same emotion → no change
  • Same cluster (e.g. peace→wonder) → crossfade only on long scenes (>20 paragraphs)
  • Different cluster (e.g. peace→tension) → always crossfade

Style Presets

Defined in lectoria/services/music.py:STYLE_PRESETS. Each preset is {include: set, exclude: set} of instrument/genre tags. A track must have ≥1 included tag AND 0 excluded tags to match.

PresetCharacteristic
autoNo filtering (default)
cinematicOrchestral, strings, soundtrack — excludes electric guitar, drums
piano_onlyPiano/keyboard — excludes most other instruments
ambientSynthesizer, pads, electronic — excludes drums, vocals
synthwaveElectronic, 80s, drum machine — excludes acoustic/classical
noir_jazzSaxophone, trumpet, jazz — excludes electronic/rock

Tag Vector Encoding

59-dimensional one-hot vector over the MTG-Jamendo mood/theme tag vocabulary. Scene vectors are built from mapping tables (EMOTION_TO_TAGS, PACING_TO_TAGS, SCENE_TYPE_TO_TAGS). Track vectors are pre-computed from their Jamendo tags. Ranking = vectorized cosine similarity (single numpy matmul over all candidates).

Image Generation

Implemented in lectoria/services/image.py. Provider: providers/image/google.py (Gemini Flash Image — uses gemini-2.5-flash-image, not Imagen).

Two Modes

Automatic (pipeline)

  • Chapter covers — from ChapterAnalysis.cover_description
  • Scene images — from Scene.image_prompt (written by LLM 2)
  • Saved to images/covers/ch{N}.png and images/scenes/ch{N}_sc{M}.png

On-demand (reader)

  • "Picture scene" button in the reader triggers POST /api/books/{id}/images/scene
  • Checks disk cache first; generates if absent
  • Reuses automatic path — so it loads automatically on revisit

Character Memory

When a scene includes a known character, the character's physical_description from LLM 1 is injected into the image prompt. If a previously generated character image exists under images/characters/, it is passed as a reference image — but only if provider.supports_reference_image() is true. Best-effort consistency, not guaranteed.

Reader Architecture

The reader paginates scenes into ~250-word pages at paragraph boundaries (utils/paginate.ts). Navigation is page-by-page with a horizontal slide animation. MusicPlayer receives chapterIndex + sceneIndex as props and fetches the appropriate track when the scene changes.

│

├── MusicPlayer      ← SSE-connected, crossfades on scene change

API Client

All API calls are in frontend/src/api/client.ts. API keys are read from localStorage on each call and injected as headers. The client uses the native EventSource API for SSE (pipeline progress). EventSource connections are closed on component unmount.

Strict TypeScript

Strict mode is enabled. No any types. Prefer interface over type for object shapes. The NCM type definitions in client.ts mirror the Pydantic models in lectoria/models/ncm.py — if the schema changes, both must be updated.

BYOK Flow

Lectoria has no server-side key storage. The complete flow:

User → Settings page → enters LLM API key
          → stored in localStorage under "llm_api_key"

User → Process Book
          → UploadPage.tsx reads localStorage["llm_api_key"]
          → sent as header: X-API-Key-LLM + X-Provider-LLM: "google"

Backend → deps.py:llm_provider_dep()
          → reads headers → calls registry.get_llm_provider("google", api_key)
          → returns GeminiLLMProvider instance (scoped to request)
          → api_key discarded after request completes

Same pattern for image generation: X-Provider-Image + X-API-Key-Image.

Security note: localStorage is vulnerable to XSS. Acceptable for an academic demo. For production: httpOnly cookies or a secrets manager. Documented limitation in thesis.

Design Decisions

These decisions appear as (Decision N) comments in the code. Full rationale in .openspec-archive/changes/multimodal-reader-system/design.md.

D1

Two-stage LLM pipeline (sequential)

LLM 1 first (full book → BookMap), then LLM 2 per chapter with BookMap context. One-directional — no iterative feedback loop. Post-processing reconciles characters discovered by LLM 2.

D2

LLM 1 receives full book in one shot

Assumes 200K+ token context window. Simpler pipeline, better character extraction. Books >700 pages are an edge case deferred.

D3

Scene boundaries via numbered paragraphs

LLM 2 returns {start_paragraph, end_paragraph} integer ranges. Validated for no gaps, no overlaps. Robust — paragraph boundaries come from EPUB HTML structure.

D4

9-category emotion taxonomy

Designed backwards from musical distinguishability, not from psychology. Categorical (not valence-arousal) because LLMs produce it reliably and it maps cleanly to Jamendo tags.

D5

LLM 2 produces image_prompt directly

No separate prompt-rewriter module. LLM 2 has full context (character descriptions, setting, scene text) to write a ready-to-use visual prompt.

D6

Music: tag filtering + cosine similarity

No ML models at read time. Emotion filter → style filter → cosine ranking of tag vectors. Fully deterministic, no CLAP, no audio embeddings.

D7

Scene-based reader (not epub.js)

Custom pagination by scene, not arbitrary visual breakpoints. Scene transitions synchronize with music changes. Loses the "book page" feel; mitigated by paginated view (D19).

D12

Music hysteresis with emotion clusters

Track changes only on meaningful emotional shifts. Four clusters (Positive/Dark/Melancholic/Neutral). Cross-cluster = always crossfade. Same-cluster = crossfade only on long scenes.

D13

BYOK provider abstraction

Python Protocol interfaces for LLM and image providers. Services never import SDK directly. New provider = new module + register call. Keys injected per-request via FastAPI dependencies.

D15

File-based JSON storage

One directory per book. No database. SQLite rejected as unnecessary for single-user, no concurrent access. NCM loaded in one read — natural access pattern.

D17

BYOK via localStorage + per-request headers

Keys stored in browser localStorage, sent as X-API-Key-* headers. Discarded server-side after use. Known XSS limitation documented in thesis.

D18

LLM output coercion + dev metadata

~40 coercion rules for hallucinated enum values. Original values in raw_* fields. ChapterAnalysis carries llm_model, attempt_count, is_fallback. Coercion is faster and cheaper than retrying for the common case.

D19

Paginated e-reader + horizontal slide

~250 words/page at paragraph boundaries. Keyboard: arrow keys. Progressive reveal forward only. Scene header only on first page; dev info on last page of scene.

D21–D23

Music style presets

Instrument/genre tags stored separately from mood tags (not merged into tag_vector). Presets = declarative {include, exclude} rules. Applied between emotion filter and cosine ranking.

D28

Gemini Flash Image (not Imagen)

Imagen 3 via generate_images requires paid billing and uses deprecated IDs. Gemini Flash Image via generate_content aligns with typical Gemini API keys.

D31–D32

CompletionResult + 429 backoff in provider

Provider returns CompletionResult(text, prompt_tokens, completion_tokens) — not plain string. Rate-limit retries (up to 3) handled at provider layer to preserve service retry budget for content failures.
